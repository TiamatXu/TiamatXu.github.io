# 查询优化 Query Optimization

在实际应用中，用户输入的问题往往比较模糊、简短或包含多重意图。直接使用原始问题进行检索 (Naive RAG) 往往效果不佳。查询优化的目标是让系统更深刻地“理解”用户想问什么。

## 多查询策略 (Multi Query)

**核心思想**：一个问题，多种问法。让大模型生成多个语义相似但表达不同的查询，并行检索，最后合并结果。

::: details e.g. 财务问答 Agent 场景
- **原始问题**：“公司去年的盈利情况怎么样？”
- **LLM 生成的变体**：
  1. “2023年度公司的净利润和毛利率是多少？”
  2. “公司2023年财报中的损益表摘要。”
  3. “公司过去一年的经营业绩分析。”
- **效果**：覆盖了问题的多个维度，大幅提高了检索的召回率。
:::

## 结果融合 (RAG-Fusion)

**核心思想**：Multi Query 的进化版。它不仅生成多个查询，还使用 **RRF (Reciprocal Rank Fusion)** 算法对并行检索回来的多个结果进行科学打分和重新排序。

**为什么 RRF 有效？**
它倾向于将那些在多个搜索结果中都排名靠前的文档置顶，避免了单一查询带来的偏差。

## 问题分解 (Decomposition)

**核心思想**：把一个复杂问题拆解成多个简单的子问题，逐个击破！

::: details e.g. 复合型财务咨询
- **用户问**：“对比北京和上海分公司在过去三年的销售额趋势，并分析原因。”
- **拆解子问题**：
  1. “北京分部过去三年的销售数据是多少？”
  2. “上海分部过去三年的销售数据是多少？”
  3. “导致两地销售额差异的主要市场因素有哪些？”
- **流程**：针对每个子问题独立检索并生成子答案，最后由 LLM 综合汇总。
:::

## 问答回退 (Step Back)

**核心思想**：不直接回答具体问题，而是先退一步生成一个更抽象、更通用的“回退问题”，从更高层次理解用户意图。

- **具体问题**：“某公司 2023 年 Q4 的流动比率是多少？”
- **回退问题**：“流动比率的计算公式及其在财务分析中的意义是什么？”
- **效果**：通过检索基础财务知识 (公式)，辅助 LLM 更准确地处理具体数据。

## 假设性文档嵌入 (HyDE)

**核心思想**：HyDE 的脑洞很大。它先让 LLM 编一个“假的答案”(虽然可能有错)，然后用这个“假设性文档”的向量去检索真实文档。

**为什么要这么做？**
因为“答案与答案”之间的相似度，往往远高于“问题与答案”之间的相似度。

```python
# HyDE 简易流程
hypothetical_answer = llm.generate("请根据常识假设性地回答：财务报表中EBITDA的调节项通常包括哪些？")
# 使用这个假设答案的向量去库里找真实的财报附注
real_docs = vectorstore.similarity_search(hypothetical_answer)
```
