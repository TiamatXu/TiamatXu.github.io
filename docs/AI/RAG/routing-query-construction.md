# 路由与查询构建 Routing & Construction

一个强大的 RAG 系统往往需要处理多种类型的数据源。路由系统就像交通指挥员，将问题送到最合适的地方。

## 智能路由 (Routing)

路由本质上是一个分类问题。根据用户问题的意图，决定使用哪种检索策略或数据源。

### 1. 逻辑路由 (Logical Routing)
利用 LLM 作为分类器，根据预定义的规则进行选择。

::: details e.g. 财务系统路由逻辑
- **用户问**：“2023年总营收是多少？” → **SQL 数据库** (查询精确数值)
- **用户问**：“公司的研发方向是什么？” → **向量数据库** (检索非结构化文本)
- **用户问**：“行业竞品近期的动态？” → **Web Search** (检索外部实时资讯)
:::

### 2. 语义路由 (Semantic Routing)
计算查询与各路由中心点（典型示例）的语义相似度，更加灵活。

## 查询构建 (Query Construction)

用户说的是“人话”，而数据库说的是“方言”。查询构建的任务是进行语言转换。

### 1. Text-to-SQL
将自然语言转换为结构化查询语句，适用于财务报表数据库。

```sql
-- 用户：找2023年毛利率超过30%的所有项目
SELECT project_name, gross_margin 
FROM finance_table 
WHERE fiscal_year = 2023 AND gross_margin > 0.3;
```

### 2. Text-to-Cypher
针对包含复杂关联关系（如：股东关系、投融资链条）的图数据库进行查询。

### 3. 混合查询构建
同时提取出语义搜索部分（用于向量检索）和结构化过滤条件（用于元数据过滤，如：日期、部门、保密级别）。

```python
# 混合查询示例
filter = {
    "publish_date": {"$gte": "2023-01-01"},
    "department": "审计部"
}
results = vectorstore.similarity_search("资产减值损失计提政策", filter=filter)
```

## 性能优化建议

1. **缓存机制**：对于常见的财务查询模式进行缓存。
2. **安全验证**：生成的 SQL 必须经过验证，防止注入风险或误执行删除操作。
3. **重试逻辑**：当生成的查询语句执行失败时，将错误反馈给 LLM 进行自我修正。
