# 多模态 RAG

> 企业文档里最有价值的信息经常不在正文里，而在**表格、图表、扫描件、架构图**中。纯文本 RAG 对这些内容要么丢弃、要么解析成乱码。**多模态 RAG** (Multimodal RAG) 让图像内容也能被检索、被理解。

## 三种技术路线

### 路线一：全部转文本

用视觉模型把图像/表格转成文字描述或 Markdown，之后走纯文本 RAG 管线：

- 表格 → Markdown 表格 (解析工具或 VLM)
- 图表 →“2021~2023 年营收分别为 X/Y/Z，呈上升趋势”式的文字摘要
- 扫描件 → OCR 文本

优点：复用现有文本管线，改造成本最低。缺点：转换必有信息损耗，复杂图表的细节问题 (“第三根柱子比第二根高多少”) 答不了。

### 路线二：多模态 Embedding

用 CLIP 类模型把图像和文本编码进**同一向量空间**，文本查询直接检索图像：

```python
# 图文同空间:文本查询“营收趋势图”能直接命中相关图表
image_vecs = clip_model.encode_images(images)
query_vec  = clip_model.encode_text("近三年营收趋势")
hits = vector_db.search(query_vec, collection="images")
```

检索到的原图交给多模态 LLM (GPT-4o、Claude、Qwen-VL) 看图作答，无转换损耗。缺点：CLIP 类模型对密集文字图表的理解有限，图文对齐质量参差。

### 路线三：视觉检索 (ColPali 路线)

跳过解析，直接把 **PDF 页面截图**作为检索单元。ColPali/ColQwen 类模型对页面截图做 token 级多向量编码 (类似 [ColBERT](./indexing-optimization) 的 Late Interaction)，查询直接匹配页面视觉特征：

- 免去 PDF 解析这个最大的脏活，表格、双栏、图文混排统统不怕
- 检索命中的整页截图交给 VLM 作答，版面信息完整保留
- 代价：多向量索引存储大，检索计算贵；2025 年以来在文档密集行业迅速流行

## 路线对比

| 维度 | 转文本 | 多模态 Embedding | 视觉检索 (ColPali) |
|:---|:---|:---|:---|
| 改造成本 | 低 (复用文本管线) | 中 | 高 (新索引 + VLM) |
| 信息保真 | 有损 | 原图无损 | 整页无损 |
| 复杂表格/版面 | ❌ 易碎 | ⚠️ 一般 | ✅ 最强 |
| 存储/计算 | 低 | 中 | 高 |

选型：文档以纯文本为主、偶有简单表格 → 路线一够用；图片素材库检索 → 路线二；财报、票据、图纸等**版面即信息**的场景 → 路线三收益最大。

## 工程要点

- **混合语料分开建索引**：文本走文本管线，页面/图像走视觉管线，检索结果在重排序层合并
- **图像也要挂元数据**：来源文件、页码、章节，引用标注和[规则过滤](./rule-based-retrieval)都靠它
- **生成端用 VLM**：检索给图，生成也要能看图——端到端多模态，任何一环退化成纯文本都会丢信息
- **评估别只测文本问答**：构建含图表问题的[评估集](./evaluation)，否则多模态能力形同虚设

::: details e.g. 财报图表问答
问题“对比图中 Q3 和 Q4 的毛利率走势”。路线一的文字摘要只有“全年毛利率上升”，答不了；路线三检索命中含该折线图的年报页面截图，VLM 直接读图：“Q3 毛利率 21.3%,Q4 升至 23.1%,斜率明显变陡”——版面即信息的场景，视觉检索是唯一解。
:::
