# 索引存储优化 Indexing Optimization

> [基础索引](./indexing)“切块即入库”有两个结构性矛盾：**检索想要小而聚焦的块，生成想要大而完整的上下文**；块被切开后**丢失了全局语境**。索引优化的各种技术，本质上都在化解这两个矛盾。

## 父子块索引 (Parent-Child / Small-to-Big)

**核心思想**：用小块检索，返回大块生成。

- 文档切成大块 (父块,~2000 token)，父块再切成小块 (子块,~400 token)
- 只对子块做向量化；检索命中子块后，**返回其父块**交给 LLM

小块语义聚焦、检索准；父块上下文完整、生成稳——一举化解第一个矛盾，是性价比最高的索引优化，多数框架 (LlamaIndex、LangChain) 开箱即用。

## 多表示索引 (Multi-representation)

**核心思想**：用摘要检索，返回原文。

为每个文档/大块生成一份聚焦核心事实的摘要，只对摘要向量化；命中摘要后返回完整原文。适合表格密集、噪声多的长文档——摘要把“年份、主体、核心指标”提炼出来，信噪比远高于直接嵌入原文。

## 上下文增强 (Contextual Retrieval)

**核心思想**：入库前给每个块补写全局语境。

块内经常出现“该公司”“同比增长 3%”这类指代，脱离原文后检索根本无从命中。Anthropic 提出的做法：入库时让 LLM 结合全文为每个块生成一句语境说明，拼在块前再向量化：

```text
原始块:  “营收同比增长 3%,主要受益于新产品线放量……”
增强后:  “[本段摘自中芯国际 2023 年年报第四章,讨论 Q4 营收表现]
          营收同比增长 3%,主要受益于新产品线放量……”
```

Anthropic 实测检索失败率降低约 49% (叠加重排序后降低 67%)，代价是索引期一次性 LLM 成本——配合 Prompt Caching 可大幅摊薄。

## 层级摘要树 (RAPTOR)

**核心思想**：递归聚类 + 逐层摘要，建一棵“抽象层级树”。

对块聚类 → 每簇生成摘要 → 对摘要再聚类再摘要，直至树根。检索时在**所有层级**同时搜索：具体问题命中叶子 (报表明细)，全局问题命中高层节点 (“近五年财务稳健性如何”)。这是向量索引对“全局性问题”的回答，与 [GraphRAG 的社区摘要](./knowledge-graph-retrieval)异曲同工。

## Token 级细粒度索引 (ColBERT)

**核心思想**：每个 token 一个向量，查询与文档逐词交互 (Late Interaction)。

相比“整块一个向量”，ColBERT 对专有名词、缩写的匹配极其敏锐，精度接近 Cross-Encoder 而速度快得多；代价是索引体积膨胀一个数量级。适合法律、医疗等术语密集且预算充足的场景。

## 选型对比

| 方法 | 化解的矛盾 | 额外成本 | 建议 |
|:---|:---|:---|:---|
| 父子块 | 检索小 vs 生成大 | 几乎为零 | 默认开启 |
| 多表示 | 长文噪声 | 索引期摘要生成 | 表格/噪声密集文档 |
| 上下文增强 | 块丢失全局语境 | 索引期 LLM 调用 | 指代密集的专业文档 |
| RAPTOR | 局部块 vs 全局问题 | 建树 + 维护复杂 | 需要全局理解的长报告 |
| ColBERT | 块粒度 vs 词粒度 | 存储 ×10 | 术语敏感的高精度场景 |

::: details e.g. 财务知识库的组合拳
年报处理管线：父子块为骨架 (子块 400 token 检索，父块 2000 token 生成) + 上下文增强 (每块前置“公司/年份/章节”语境) + 表格章节额外做摘要索引。三种技术叠加，指代类问题 (“该季度环比如何”) 的检索命中率提升立竿见影。
:::
