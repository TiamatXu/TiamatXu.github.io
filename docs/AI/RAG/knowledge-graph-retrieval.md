# 基于知识图谱的关系检索

> **知识图谱检索** (Knowledge Graph Retrieval) 不再把文档当作孤立的文本块，而是抽取出实体与关系构成图结构，通过图遍历回答“关系型”与“多跳”问题，代表方案是 **GraphRAG**。

## 为什么需要图谱检索

向量检索的单位是“文本块”，天然回答不了跨文档的关系问题：

- “A 公司和 B 公司之间有哪些共同投资人？”—— 答案分散在多个文档中，任何单块都不完整
- “这个数据集的主要主题是什么？”—— 全局性总结问题，Top-K 文本块无法覆盖全貌

知识图谱把分散的事实连接成网络，检索从“找相似文本”变成“沿关系遍历”。

## 核心流程

1. **实体与关系抽取**：用 LLM 从文档中抽取三元组 (主体，关系，客体)，如 `(马斯克, 创立, SpaceX)`
2. **图构建**：三元组入图数据库 (Neo4j 等)，同名实体消歧合并
3. **社区检测与摘要** (GraphRAG 特有)：对图做聚类，为每个社区预生成摘要，服务全局性问题
4. **图遍历检索**：查询解析出起点实体，沿边多跳扩展，收集子图作为上下文

## 工程实现要点

```cypher
// Neo4j: 查询与「张三」二跳以内的所有公司
MATCH (p:Person {name: "张三"})-[*1..2]-(c:Company)
RETURN DISTINCT c.name
```

- **抽取质量是天花板**：LLM 抽取三元组的准确率决定整个图谱的可用性，需要对 schema (允许哪些实体/关系类型) 做约束
- **Local vs Global 查询**：GraphRAG 区分局部查询 (从实体出发遍历) 和全局查询 (基于社区摘要归纳)，路由错了效果会很差
- **构建成本高**：全量文档过一遍 LLM 抽取，token 成本远超向量索引，适合高价值、关系密集的语料
- **与向量检索混用**：实践中常用向量检索定位入口实体/文本块，再用图遍历扩展关系上下文

## 方案对比

| 维度 | 向量 RAG | GraphRAG |
|:---|:---|:---|
| 简单事实问题 | ✅ 快、便宜、够准 | ⚠️ 大材小用 |
| 多跳关系问题 | ❌ 文本块覆盖不到 | ✅ 沿边遍历直达 |
| 全局总结问题 | ❌ Top-K 无法覆盖全貌 | ✅ 社区摘要 |
| 索引构建成本 | 低 | 高 (LLM 全量抽取) |
| 数据更新 | 简单 (增量向量化) | 复杂 (需增量抽取与消歧) |

::: details e.g. 金融风控场景
排查“某贷款申请人是否与已知欺诈团伙有关联”：申请人 → 手机号 → 关联账户 → 历史欺诈标记，这是典型的三跳关系查询。向量检索对此无能为力，图谱检索沿边遍历即可给出可解释的关联路径。
:::
