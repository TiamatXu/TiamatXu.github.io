# 高级 RAG 架构 Sophisticated RAG

当传统的“检索 + 生成”模式遇到低质量文档或检索偏差时，容易产生幻觉。高级架构通过引入“闭环反馈”来解决这一问题。

## 纠错检索增强生成 (CRAG)

**核心思想**：给 RAG 加一个“质检员”。在把结果交给 LLM 之前，先评估检索内容的可靠性。

### CRAG 的三种处理路径：
1. **Correct (正确)**：检索内容非常相关，进行“知识精炼”后直接生成答案。
2. **Incorrect (错误)**：检索内容完全不相关，抛弃该结果，触发 **网络搜索 (Web Search)**。
3. **Ambiguous (模糊)**：检索内容半对半错，结合“内部文档”和“网络搜索”进行信息融合。

::: details e.g. 财务合规检查场景
- **场景**：用户询问某项最新的地方性财务补贴政策。
- **流程**：CRAG 发现本地数据库只有 2022 年的旧政策（标记为 Incorrect），于是自动触发网络搜索查找 2024 年的最新公告，确保回答的时效性。
:::

## 自反思检索增强生成 (Self-RAG)

**核心思想**：让系统学会“三省吾身”。模型在生成过程中不断通过特殊的 **反思令牌 (Reflection Tokens)** 自我提问。

### Self-RAG 的自省维度：
- **我需要检索吗？**：如果问题是常识（如“什么是资产负债表？”），直接回答；如果是事实性数据，则触发检索。
- **检索到的文档相关吗？**：评估检索结果的质量。
- **我生成的内容有事实支持吗？**：确生成的每一句话都能在文档中找到依据。
- **生成的答案有用吗？**：评估答案是否真正解决了用户的问题。

## 架构对比

| 特性 | 传统 RAG | CRAG | Self-RAG |
| :--- | :--- | :--- | :--- |
| **是否总是检索** | 是 | 是 | 动态决策 (按需) |
| **评估机制** | 无 | 评估检索质量 | 评估检索 + 生成质量 |
| **纠错能力** | 无 | 触发网络搜索补救 | 循环迭代直到满意 |
| **复杂度** | 低 | 中 | 高 (需微调或复杂提示词) |

::: details e.g. 财务 Agent 的 Self-RAG 表现
在处理一份复杂的“跨年度财务数据对比”任务时，Self-RAG 会先检索数据，生成草稿后发现某些年份数据对不上，于是标记为“支持度低”，重新发起精准检索并修正答案，最后输出一份逻辑严丝合缝的分析报告。
:::
