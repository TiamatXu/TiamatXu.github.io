# 高级 RAG 架构 Sophisticated RAG

> 固定管线的 RAG 有个致命假设：“检索回来的东西总是可用的”。现实并非如此。高级架构给 RAG 加上**质量评估与纠错闭环**，并进一步把“是否检索、检索几次”的决策权交给模型——从固定流程演进为动态系统。

## 纠错检索 (CRAG)

**核心思想**：在生成之前给检索结果加一道“质检”。轻量评估器给检索内容打分，按结果分流：

1. **Correct**：相关性高 → 知识精炼后直接生成
2. **Incorrect**：完全不相关 → 抛弃，触发 Web 搜索补救
3. **Ambiguous**：半对半错 → 内部文档与 Web 搜索融合

::: details e.g. 财务合规检查场景
用户询问最新的地方性财务补贴政策，CRAG 发现本地库只有 2022 年旧政策 (判为 Incorrect)，自动转 Web 搜索获取 2026 年最新公告——固定管线会拿旧政策硬答，CRAG 用纠错路径保住了时效性。
:::

## 自反思检索 (Self-RAG)

**核心思想**：模型在生成过程中用**反思令牌** (Reflection Tokens) 持续自问：

- 这个问题需要检索吗？(常识直答，事实才检索)
- 检索到的文档相关吗？
- 我生成的每句话有证据支持吗？
- 答案真的解决了问题吗？

任何一问不过关就重新检索或重写，循环直到自评通过。能力最强，但需要专门训练或复杂提示词工程，落地成本最高。

## Agentic RAG

**核心思想**：把检索当作 [Agent 的一个工具](/AI/agent/tool-use)。LLM 不再位于检索的下游，而是主导整个过程：

```text
Agent 循环:分析问题 → 决定检索什么 → 执行检索 → 评估结果够不够
           → 不够?改写查询再检索 / 换数据源 → 够了?生成答案
```

与 CRAG/Self-RAG 的“预设纠错路径”不同，Agentic RAG 的行为完全由模型运行时决策——多跳问题自动多轮检索，歧义问题先澄清再检索。它复用 Agent 的全套机制：[ReAct 循环](/AI/agent/react-planning)、工具调用、多数据源路由。

> 成本警告：Agentic RAG 的质量提升只在困难、多跳、歧义问题上兑现；简单事实问题走 Agent 循环纯属浪费。**先把混合检索 + 重排序的底座做扎实，再考虑 Agentic 化**——这是 2026 年生产实践中最常被违反的一条共识。

## Adaptive RAG

**核心思想**：不是所有问题都配用重管线。前置一个查询分类器，按复杂度分流：

- 简单事实问题 → 单次向量检索，甚至不检索
- 中等问题 → 混合检索 + 重排序
- 复杂多跳问题 → Agentic RAG

用便宜的分类调用，省下大量重管线开销，是[路由](./routing-query-construction)思想在“管线复杂度”维度上的推广。

## 架构对比与选型

| 特性 | 传统 RAG | CRAG | Self-RAG | Agentic RAG | Adaptive RAG |
|:---|:---|:---|:---|:---|:---|
| 是否总是检索 | 是 | 是 | 按需 | 模型决策 | 按复杂度分流 |
| 纠错能力 | 无 | Web 搜索补救 | 反思循环 | 自主重试/换源 | 取决于所选管线 |
| 延迟/成本 | 低 | 中 | 高 | 高且波动 | 平均最优 |
| 落地难度 | 低 | 中 | 高 | 中 (框架成熟) | 中 |

选型路径：固定管线 (混合检索 + 重排序) → 时效性/纠错需求上 CRAG → 多跳复杂问题占比高再上 Agentic → 流量大、问题难度分布宽，用 Adaptive 做总调度。每一步升级都应有[评估数据](./evaluation)支撑，而不是追新。
