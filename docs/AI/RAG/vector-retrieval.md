# 基于向量的语义检索

> **向量检索** (Vector Retrieval) 将文本映射为高维向量 (Embedding)，通过计算向量间的距离来衡量语义相似度，是 RAG 中最主流的检索方式。

## 核心原理

1. 向量化：用 Embedding 模型将文档块和查询编码为高维向量 (常见 768~3072 维)
2. 索引构建：将文档向量存入向量数据库，构建近似最近邻 (ANN) 索引
3. 相似度计算：查询向量与文档向量比较，返回 Top-K 最相似结果

### 相似度度量

| 度量方式 | 计算 | 适用场景 |
|:---|:---|:---|
| **余弦相似度** | 向量夹角 | 文本语义检索的默认选择，不受向量长度影响 |
| **点积** | 内积 | 归一化向量下等价于余弦，计算更快 |
| **欧氏距离** | 直线距离 | 对向量长度敏感，图像等稠密特征常用 |

### ANN 索引算法

暴力检索 (Flat) 精度最高但复杂度线性增长，生产环境用近似算法换取速度：

- **HNSW**：分层可导航小世界图，查询快、召回高，内存占用大，是当前事实标准
- **IVF**：倒排文件索引，先聚类再在簇内搜索，适合超大规模、内存受限场景
- **PQ (乘积量化)**：压缩向量降低内存，常与 IVF 组合 (IVF-PQ)，牺牲少量精度

## 工程实现要点

```python
import chromadb

client = chromadb.Client()
collection = client.create_collection(
    name="docs",
    metadata={"hnsw:space": "cosine"}  # 指定余弦相似度
)

collection.add(documents=["RAG 结合检索与生成...", "向量数据库存储..."], ids=["1", "2"])
results = collection.query(query_texts=["什么是检索增强生成？"], n_results=3)
```

- **Embedding 模型选型**：优先看 MTEB / C-MTEB (中文) 榜单，兼顾维度、延迟与成本
- **查询与文档不对称**：问句和陈述句分布不同，选支持指令前缀的模型 (如 BGE 的 `query:` / `passage:`)
- **Top-K 与阈值**：只设 Top-K 会引入不相关结果，配合相似度阈值过滤更稳
- **向量归一化**：使用点积前先归一化，否则长文档向量会占优

## 优势与局限

| 维度 | 表现 |
|:---|:---|
| 语义理解 | ✅ 同义改写、跨语言表述也能命中 |
| 精确匹配 | ❌ 型号、错误码、人名等专有词容易漏检 |
| 可解释性 | ❌ 相似度分数难以向用户解释 |
| 更新成本 | ⚠️ 换 Embedding 模型需全量重建索引 |

::: details e.g. 语义命中 vs 精确漏检
用户搜“电脑开不了机怎么办”，向量检索能命中“主机无法启动的排查步骤”(语义相近)；但搜“Error 0x80070057”时，向量检索可能返回一堆“Windows 报错处理”的泛泛文档，反而不如关键词检索直接命中该错误码。这也是生产环境普遍采用“向量 + 关键词”混合检索的原因。
:::
