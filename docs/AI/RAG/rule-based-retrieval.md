# 基于规则的过滤检索

> **规则过滤检索** (Rule-based / Filtered Retrieval) 在相似度计算之前或之中，先用确定性的规则 (元数据条件、权限、时间范围) 收窄候选集，保证检索结果“合规、可控、可预期”。

## 为什么需要规则过滤

纯相似度检索有两个生产环境无法接受的问题：

- **越权**：员工 A 的问题命中了员工 B 部门的机密文档 —— 相似度再高也不能返回
- **失效**：2022 年的旧政策与 2026 年的新政策语义几乎一样，相似度无法区分“哪个还有效”

这些是**确定性约束**，必须用规则表达，不能交给概率化的相似度。

## 常见规则维度

| 维度 | 示例 | 典型来源 |
|:---|:---|:---|
| 权限 | `department = "finance"` | 用户身份、RBAC 系统 |
| 时效 | `effective_date >= 2026-01-01` | 文档元数据 |
| 来源 | `source IN ("官方文档", "内部 Wiki")` | 入库时标注 |
| 文档类型 | `type = "contract"` | 文件格式/分类模型 |
| 语言/地区 | `lang = "zh-CN"` | 检测或标注 |

## 工程实现要点

```python
results = collection.query(
    query_texts=["报销标准是多少？"],
    n_results=5,
    where={  # 元数据预过滤
        "$and": [
            {"department": {"$eq": "finance"}},
            {"year": {"$gte": 2025}},
        ]
    },
)
```

- **预过滤 vs 后过滤**：预过滤 (先筛再检索) 结果数量可控，是主流；后过滤 (检索完再筛) 可能把 Top-K 筛到空
- **元数据在入库时决定一切**：分块时就要把部门、日期、来源等字段挂到每个 chunk 上，事后补标注成本极高
- **权限过滤必须在服务端强制**：不能依赖提示词让 LLM“自觉”不引用越权内容，过滤要发生在检索层
- **规则来自查询构建**：配合[路由与查询构建](./routing-query-construction)，LLM 可以把自然语言“去年财务部的报销制度”解析成结构化过滤条件

## 与其他检索策略的关系

规则过滤不是独立的第四种检索，而是叠加在向量/关键词/图谱检索之上的**约束层**：

```text
用户查询 → 解析过滤条件 → [规则收窄候选集] → 向量/BM25 检索 → 重排序 → 生成
```

::: details e.g. 多租户 SaaS 知识库
一个服务多家企业的客服机器人，每次检索都必须带上 `tenant_id = 当前租户` 的硬过滤。哪怕两家企业的文档内容高度相似，也绝不允许跨租户返回 —— 这类隔离靠相似度阈值是做不到的，只能靠规则层强制。
:::
