# ReAct 与规划

> **ReAct** (Reasoning + Acting) 让 LLM 交替进行“推理”和“行动”：先想清楚下一步，再执行工具调用，观察结果后继续推理。它是 Agent 的基础运行范式；**规划** (Planning) 则解决 ReAct 在复杂长任务上“走一步看一步”的短视问题。

## ReAct 循环

```text
Thought:  用户想比较两款手机的续航，我需要分别查询参数
Action:   search("iPhone 17 电池容量")
Observation: 3800mAh，视频播放 26 小时
Thought:  拿到第一款了，再查第二款
Action:   search("小米 16 电池容量")
Observation: 5500mAh，视频播放 31 小时
Thought:  两款数据齐了，可以对比作答
Final Answer: 小米 16 续航更强……
```

三要素缺一不可：

- **Thought (思考)**：显式写出推理过程，让决策可追溯、可调试
- **Action (行动)**：从工具集中选择一个并给出参数
- **Observation (观察)**：工具的真实返回，作为下一轮思考的依据

> ReAct 的关键价值在于**用真实观察对齐推理**：纯推理 (Chain-of-Thought) 会在缺少事实时编造，纯行动会盲目试错，交替进行则让每一步推理都踩在事实上。

## ReAct 的局限

- **短视**：每次只决策一步，长任务容易绕路、重复、忘记最终目标
- **错误累积**：中间某步观察被误读，后续全部跑偏
- **成本不可控**：循环次数由模型决定，可能远超预期

## 规划模式

### Plan-and-Execute

先让模型一次性生成完整计划，再逐步执行：

```text
Plan:
  1. 查询 A 公司近三年营收
  2. 查询 B 公司近三年营收
  3. 计算增长率并对比
  4. 生成对比报告
Execute: 按步骤执行，每步完成后可触发重规划 (Re-plan)
```

- 优点：全局视野、步骤可并行、进度可展示给用户
- 缺点：计划可能赶不上变化，需要“执行失败 → 重规划”的闭环

### 任务分解 (Task Decomposition)

把大目标拆成可独立验证的子任务，常配合任务清单 (Todo List) 持久化——这也是 Claude Code、Cursor 等编码 Agent 的标准做法：模型每完成一项就勾掉一项，防止长会话中目标漂移。

## 模式选型

| 任务特征 | 推荐模式 |
|:---|:---|
| 短任务 (3~5 步以内)、路径不确定 | 纯 ReAct |
| 长任务、步骤可预估 | Plan-and-Execute + 重规划 |
| 超长任务、需要断点续跑 | 任务分解 + 持久化任务清单 |

::: details e.g. 编码 Agent 的规划实践
“给项目加用户登录功能”这类任务，直接 ReAct 会写到哪算哪。成熟做法是先分解：① 设计数据表 ② 写注册接口 ③ 写登录接口 ④ 加中间件 ⑤ 补测试——每完成一项跑一次测试验证后再继续，任何一步失败只重做该步，而不是从头再来。
:::
