# 上下文工程

> **上下文工程** (Context Engineering) 研究的问题是：在每一步决策时，让上下文窗口里恰好装着“做对这一步所需的最小信息集”。它在 2025 年后取代提示词工程成为 Agent 开发的核心学科——提示词工程优化“怎么说”，上下文工程管理“模型看见什么”。

## 为什么上下文是稀缺资源

上下文窗口再大，**有效注意力**也是有限的：

- **Context Rot**：随着 token 数增长，模型对窗口中部内容的召回率明显下降，长上下文不等于好上下文
- **成本线性增长**：每轮循环都携带全部历史，token 费用和延迟随任务长度爆炸
- **噪声稀释决策**：无关的工具返回、过期的中间结果，都在和关键信息抢注意力

Agent 任务动辄几十轮工具调用，上下文管理不当的直接后果是：任务后半程“忘记目标”“重复已做过的事”。

## 上下文的构成与预算

```text
┌─ System Prompt:角色、规则、工具说明     (相对固定,可缓存)
├─ 工具定义:JSON Schema                  (相对固定,可缓存)
├─ 记忆/知识:检索到的文档、长期记忆       (按需注入)
├─ 对话历史:用户消息、模型思考            (随轮次增长)
└─ 工具结果:每轮 Action 的 Observation   (增长最快的部分)
```

两条铁律：**稳定的前缀放前面** (命中 KV Cache，省钱提速)；**增长最快的部分优先治理** (工具结果是头号膨胀源)。

## 四类核心技术

### 压缩 (Compaction)

接近窗口上限时，把旧历史摘要成一段“进度纪要”，保留目标、关键决策、未完成事项，丢弃原始工具返回。Claude Code 的 auto-compact 就是这个机制。

```python
if token_count(history) > 0.8 * window:
    summary = llm.invoke(f"把以下执行历史压缩为进度纪要,保留:目标、已完成、关键结论、待办:\n{history}")
    history = [system_prompt, summary, *recent_turns(3)]  # 最近几轮保留原文
```

### 检索式上下文 (Just-in-Time)

不把知识全塞进窗口，而是给 Agent 一个检索工具，用到什么查什么——本质上是把 [RAG](/AI/RAG/introduction) 用作上下文管理手段。文件路径、轻量标识符留在上下文里，重内容按需加载。

### 结构化外部笔记

让 Agent 把状态写到窗口之外：任务清单 (todo)、进度文件、[持久化记忆](./memory)。上下文被压缩或截断后，这些外部状态仍在，是长任务防“目标漂移”的关键——[规划](./react-planning)篇的任务清单就是最常用的形态。

### 子智能体隔离

把高噪声的子任务 (大范围搜索、日志分析) 丢给[子 Agent](./multi-agent)，它在自己的独立上下文里烧掉几万 token，只把几百 token 的结论带回主上下文——上下文隔离是多智能体架构最实际的动机。

## 工具结果的瘦身

头号膨胀源要单独治理：

- 工具返回**只带决策需要的字段**，不要整包 JSON 原样回传 (见[工具调用](./tool-use)的设计原则)
- 大结果落盘，返回“文件路径 + 摘要”，需要细节时再读
- 已被后续步骤消化的旧 Observation，可在压缩时最先丢弃

## 反模式对照

| 反模式 | 正确做法 |
|:---|:---|
| 把整个知识库塞进 System Prompt | 检索式按需注入 |
| 工具返回原样进上下文 | 字段裁剪 + 大结果落盘 |
| 历史无限追加直到窗口爆掉 | 阈值触发压缩,外部笔记兜底 |
| 每轮动态改写 System Prompt | 稳定前缀 + 缓存,动态内容放消息尾部 |

::: details e.g. 一次代码重构任务的上下文管理
任务跨 60 轮工具调用。System Prompt 与工具定义全程不动 (缓存命中)；todo 文件记录 12 个子任务的完成状态；每次全文件搜索交给子 Agent，主上下文只收结论；第 35 轮触发压缩，前 30 轮历史变成 300 token 纪要。最终第 60 轮时模型依然清楚知道“在做什么、做到哪了”——没有这套机制，第 40 轮就开始重复改同一个文件。
:::
