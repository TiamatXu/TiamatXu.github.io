# 记忆机制

> 上下文窗口是 Agent 的“工作台”，会话结束就清空。**记忆** (Memory) 让 Agent 跨会话积累知识：用户偏好、项目约定、过往结论——没有记忆的 Agent 每次都是“新来的实习生”，有记忆的 Agent 才谈得上“越用越懂你”。

## 记忆的分层模型

业界 (Letta/MemGPT 谱系) 的共识是三层结构，类比操作系统的存储层级：

| 层级 | 位置 | 内容 | 类比 |
|:---|:---|:---|:---|
| **核心记忆** (Core) | 始终在上下文内 | 用户画像、当前任务要点,极小且高价值 | 寄存器/内存 |
| **召回记忆** (Recall) | 窗口外,可检索 | 完整对话历史 | 磁盘上的日志 |
| **归档记忆** (Archival) | 窗口外,可检索 | 沉淀后的事实、偏好、结论 | 数据库 |

核心思想：**Agent 自己管理记忆分页**——像操作系统换页一样，把重要信息从窗口“写出去”，需要时再“调进来”。这是[上下文工程](./context-engineering)在时间维度上的延伸。

## 记忆的两个基本操作

### 写入：决定什么值得记

写入质量决定记忆系统的上限，原则是**记结论，不记流水账**：

- 值得写：用户纠正过的偏好 (“commit message 用中文”)、稳定事实 (“项目用 pnpm”)、踩坑结论
- 不值得写：一次性上下文、能从代码/文档重新推导的信息、时效性内容不转成绝对时间就入库

```python
memory_tool = {
    "name": "save_memory",
    "description": "保存跨会话有价值的事实。只保存:用户偏好、稳定的项目约定、"
                   "重要结论。不要保存一次性信息。写入前先检查是否与已有记忆重复。",
    "input_schema": {"type": "object", "properties": {
        "content": {"type": "string"}, "category": {"type": "string"}}},
}
```

### 读取：记忆检索就是 RAG

会话开始或任务切换时，按当前语境检索相关记忆注入上下文——技术栈与 [RAG 检索](/AI/RAG/retrieval)完全同构 (向量 + 关键词 + 重排序)，语料换成了记忆条目。

## 三种实现路线

- **文件式**：记忆就是 Markdown 文件 (如 CLAUDE.md、记忆目录)，Agent 用读写文件工具自助管理。最简单、可人工审阅，Claude Code 等编码 Agent 的主流做法
- **记忆服务**：Mem0、Zep (Graphiti) 等专用层，自动做抽取、去重、冲突合并，Zep 还把记忆组织成时序知识图谱，擅长“用户三月说 A、五月改口 B”类时间推理
- **框架内建**：Letta 的自编辑记忆、LangGraph 的 LangMem/Store，与编排框架深度耦合

选型：单人工具 → 文件式；多用户产品 → 记忆服务；已重度使用某框架 → 框架内建。

## 工程要点

- **记忆会过期**：注入时带上写入时间，让模型自行判断时效；定期让 Agent 复核冲突条目
- **去重与合并**：同一事实反复写入会稀释检索质量，写入前先查重，新事实与旧记忆矛盾时更新而非追加
- **隔离边界**：多用户系统的记忆必须按 user_id 硬隔离——这是[规则过滤](/AI/RAG/rule-based-retrieval)的直接应用，也是安全红线
- **评估**：LoCoMo、LongMemEval 是记忆系统的主流基准，时间推理和多跳问题是当前拉开差距的地方

::: details e.g. 编码 Agent 的记忆生长过程
第 1 次会话：用户纠正“测试要用 vitest 不是 jest”→ 写入归档记忆。第 5 次会话：任务涉及测试，检索命中该条目，Agent 直接用 vitest，无需再被纠正。第 20 次会话：项目迁移到 bun test,Agent 发现记忆与现实冲突，更新旧条目——记忆不是只增的日志，是需要维护的知识库。
:::
