# 微服务与 RPC 高频

## 分布式理论：CAP 与 BASE

:::info 知识点
一致性、可用性、分区容错、最终一致性
:::

* **CAP 定理**：
    * **C (Consistency)**：强一致性。所有节点同时读到最新数据；
    * **A (Availability)**：可用性。服务必须能在正常时间内响应；
    * **P (Partition tolerance)**：分区容错。网络分区时系统仍能运行；
    * **核心结论**：三者不可兼得，P 必须保证，系统需在 CP (金融级) 或 AP (社交级) 间权衡。
* **BASE 理论**：CAP 的延伸。
    * **Basically Available**（基本可用）；
    * **Soft state**（软状态）；
    * **Eventually consistent**（最终一致性）。

## 注册中心 Nacos 的 CP/AP 模型

:::info 知识点
选型配置、隔离级别
:::

* **AP 模式**：默认。强调高可用，允许短暂的数据不一致。适合大规模服务发现；
* **CP 模式**：强调强一致性。若发生分区，可能会停止服务注册直到一致。适合对数据准确性要求极高的场景。

## 一致性算法：Gossip、Paxos、Raft

:::info 知识点
分布式共识、Leader 选举、日志复制
:::

* **Gossip**：类似“病毒传播”或“八卦”。去中心化、最终一致性，适合大规模集群节点状态同步（如 Redis Cluster）；
* **Paxos**：分布式一致性的鼻祖。理论完善但实现极其复杂；
* **Raft**：Paxos 的简化版。通过 **Leader 选举**、**日志复制**、**安全性**三个子问题实现强一致性。广泛用于 ETCD、Consul。

## 微服务高可用：限流、熔断、降级、隔离

:::info 知识点
服务容错、熔断状态机、哨兵模式
:::

* **限流**：限制并发请求量（漏桶/令牌桶算法），防止服务被压垮；
* **熔断**：类似“保险丝”。当下游服务故障率达到阈值，直接切断调用，返回错误或默认值（状态：Open, Half-Open, Closed）；
* **降级**：牺牲非核心功能（如返回空数据），保证核心业务运行；
* **隔离**：资源隔离（线程池/信号量隔离），防止因某个接口故障拖垮整个系统线程池。
