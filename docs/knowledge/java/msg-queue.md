# MQ 消息队列高频

## 如何选择 Kafka、RabbitMQ、RocketMQ？

:::info 知识点
选型依据、吞吐量、可靠性、延时
:::

* **Kafka**：**极致吞吐量**（百万级）。适合大数据日志采集、实时计算。单机高吞吐，但延时稍高（毫秒级）；
* **RabbitMQ**：**极致可靠性/灵活性**。基于 Erlang，支持丰富路由，延时极低（微秒级）。吞吐量较低（万级）；
* **RocketMQ**：**金融级一致性**。阿里开源，功能丰富（事务消息、延时队列）。性能与可靠性的平衡（十万级吞吐）。

## 消息队列如何保证消息不丢失？

:::info 知识点
生产端确认、存储端持久化、消费端 ACK
:::

* **生产者**：配置确认机制（如 RabbitMQ 的 Confirm、Kafka 的 acks=all）；
* **MQ 集群**：开启磁盘持久化，镜像队列或多副本备份；
* **消费者**：关闭自动 ACK，处理完业务后再手动确认。

## 如何处理消息堆积？

:::info 知识点
扩容、临时分流、优化逻辑
:::

* **增加消费者**：最直接。需确保 Topic 分区数 >= 消费者实例数；
* **优化逻辑**：排查消费端是否存在耗时操作（如 I/O 阻塞），改用线程池并发处理；
* **临时分流**：新建 Topic 和分区，将旧 Topic 消息转发到新队列，利用更多机器抗压。

## 什么是死信队列 (DLX)？

:::info 知识点
拒绝签收、超时过期、队列溢出
:::

* **定义**：消息无法被正常消费时，被 MQ 重新投递到的特定队列。
* **产生原因**：
    1. 消费者拒绝确认（basic.nack/reject）且未重回队列；
    2. 消息 TTL 过期；
    3. 队列达到最大长度。

## Kafka 的高性能秘诀

:::info 知识点
顺序写、零拷贝、批量发送
:::

* **顺序读写**：直接在磁盘文件末尾追加数据，避免随机寻址开销；
* **Page Cache**：利用内核操作系统的页缓存；
* **零拷贝 (Zero-copy)**：利用 `sendfile` 技术，数据从 Page Cache 直接发往网卡，减少用户态拷贝。
